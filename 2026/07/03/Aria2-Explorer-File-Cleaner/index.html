Aria2 进阶：如何安全地为 Aria2-Explorer 扩展增加「删除任务并清理本地文件」功能

发表于 2026-07-03 分类于 实战教程 阅读次数：

在大数据与 PT 下载的时代，Aria2 凭借其轻量级、无图形界面低资源消耗，以及强大的 RPC 远程调用能力，成为了无数 NAS 玩家、Home Lab 爱好者以及下载狂热者的“万金油”神器。

然而，凡是使用过 Aria2 的朋友，几乎都遇到过一个极其头疼的痛点：在前端网页或浏览器扩展（如 Aria2-Explorer、AriaNg）中点击“删除任务”，仅仅是将该任务从 Aria2 的下载队列中抹去。原本下载在本地磁盘的大文件、分块碎片，以及伴随的 .aria2 控制文件，依然纹丝不动地躺在硬盘里。 日积月累，磁盘空间悄无声息地被这堆“幽灵文件”吞噬殆尽。

虽然该需求在 Aria2-Explorer 的 GitHub Issue 列表中高悬已久（例如经典的 #52「求添加删除任务时同时删除对应文件功能」），但由于各平台权限限制，官方至今没有提供原生支持。

今天，我们将从 Full-Stack 安全开发与系统防提权 的视角出发，彻底打通浏览器扩展与本地操作系统的壁垒，实现一套极度安全、防止命令注入、界面不卡死的本地文件自动清理方案。

第一部分：架构破局 —— 为什么需要本地 Helper 服务？

1. 浏览器的“楚河汉界”：沙箱限制

现代浏览器扩展（WebExtension）运行在一个被严格隔离的沙箱（Sandbox）环境中。即使我们在前端拿到了下载文件的绝对路径，扩展也绝对无法直接调用操作系统的裸文件系统接口（File System API）去执行物理删除。这在安全策略上被称为“最小权限原则”，防范恶意扩展随意抹除用户磁盘。

2. 破局思路：本地轻量级桥梁

既然扩展出不去，操作系统进不来，我们就在本地架设一座安全的桥梁。

通过这套异步链路，我们在删除任务的瞬间完成本地文件和临时控制文件的精准清理，随后再将删除指令送达 Aria2 服务。

3. 前端改造：放弃油猴，直攻反混淆源码

常见的用户脚本（油猴）在 chrome-extension:// 协议下的内部页面注入极不稳定。因此，我们选择直接对 Aria2-Explorer 扩展的本地前端代码下手。
由于扩展的编译产物 aria-ng-xxx.min.js 是高度混淆压缩的，直接打正则补丁不仅难以调试，且极易在升级时引入隐蔽 Bug。我们使用 js-beautify 将其彻底反混淆还原为可读版本，直接在格式化后的代码上插入菜单模板、绑定 UI 控制器并拦截 RPC 交互。

第二部分：安全防线 —— 如何让本地 Helper 监控服务坚不可摧？

本地运行一个具有文件删除权限的 HTTP 服务（通常长驻于 127.0.0.1:6801）是极其危险的。一旦防护不当，任何恶意网页均可以通过跨域网络请求（CORS）或者接口扫描，提权抹除您的整块硬盘。

2. 严格的 CORS 域白名单审查

本地服务绝不能对所有人开放。除了最基础的 OPTIONS 预检拦截外，服务强制验证请求头中的 Origin 属性。

任何源自非白名单扩展或普通网页的跨域 Fetch 请求，都会被浏览器和 Helper 在预检阶段直接抛出 403 Forbidden 并记录拦截日志。

3. 规范化路径白名单限制（防止目录遍历）

攻击者可能传入如 ../../Windows/System32 之类的相对路径。我们通过 path.resolve 对路径强制进行绝对路径解析，并校验其前缀是否属于被允许的下载目录白名单：

4. 命令注入防御与“安全移至回收站”

许多初学者在执行系统级删除时会直接调用 child_process.exec('rm -rf ...')，这极易引入命令注入漏洞（例如文件名中带有分号 ; 或 &）。
我们通过使用 PowerShell 的 单引号字符串（'{path}'）闭包来传递参数，并对文件名中的所有单引号进行双单引号逃逸（replace(/'/g, "''")），同时引入 Windows Shell 原生 API 将文件移入系统回收站（Recycle Bin），既防止了提权注入，又留有后悔药：

在操作审计日志 operation.log 中，我们记录请求被拦截的事件，但绝不打印请求体中携带的 Token 明文，确保日志文件的合规安全性。

第三部分：踩坑实录 —— 排查并解决“正在加载”卡死天坑

在将方案前端接入的实际测试中，我们遇到了一个经典悬挂 Bug：点击删除后，界面会永久卡在 AriaNg 自带的“正在加载…”半透明遮罩层中。

1. tellStatus 的“第二参数”天坑

为了在文件被销毁前获取它，我们在服务层拦截了 removeTasks，并调用了 RPC 获取最新文件列表：

根因剖析：AriaNg 服务框架内部，i.tellStatus(options, isMulticall) 函数的第二个参数表示“是否为批量 multicall 构造元素”。当传入 true 时，该方法仅仅返回封装好的请求 JSON 结构，根本不会真正建立网络请求发送出去！因此回调永远不会执行，Promise 链无限挂起，遮罩层彻底卡死。
解决方案：直接移除第二个参数 true。

这意味着遮罩层的显隐完全依赖于挂载在 Scope 上的 loadPromise 什么时候被 Settle（即被 Resolve 或 Reject）。

3. BT 多文件与 .aria2 控制文件清除逻辑

  • BT 多文件：在多文件任务中，task.files 会列出所有具体的文件路径，我们在清单中同时推入这些具体路径与它们共同的父下载文件夹：task.dir + "/" + task.taskName。Helper 将对该文件夹进行递归移入回收站处理。
  • .aria2 控制文件：无论单文件下载还是 BT 下载，Aria2 均会在下载目录下创建一个以文件/文件夹命名的同名控制文件。我们一并推入 task.dir + "/" + task.taskName + ".aria2" 进行销毁，完美解决垃圾碎片的残留。

第四部分：自启动部署与健康状态检测

为了让这套服务在 Windows 启动时静默长驻后台，我们为其配置了无窗口启动机制。

1. 静默自启动脚本配置

在 Windows 系统中，新建一个后缀为 .vbs 的脚本，存放在开机启动文件夹中（可使用快捷键 Win+R 输入 shell:startup 快速打开）：

start-aria2-file-helper.vbs

注：这里的参数 0 代表隐藏控制台黑窗口，false 表示异步执行不阻塞系统开机。
